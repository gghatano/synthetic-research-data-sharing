# タスク粒度ガイド

1 Issue の粒度は **Claude Code が 1 セッションで理解・実装・検証・説明できる範囲**。
大きすぎる Issue は仕様解釈・実装・テスト・設計判断を同時に抱え、品質が落ちる。

## 1 Issue が満たすべき条件
- 1 つの PR で完結する
- 変更対象が過度に広くない
- 完了条件が明確である
- テストまたは代替検証が可能である
- 他 Issue への依存関係が明示されている
- 失敗時に切り戻しやすい

## 粒度の例（適切 / 不適切）

| 粒度 | 適切な Issue 例 | 不適切な Issue 例 |
|------|----------------|-------------------|
| 調査 | 既存の KM 実装が Phoenix 基準をどう扱うか確認する | 生存分析をよしなに改善する |
| 基盤 | pytest 基盤と最初の再現性テストを追加する | テスト環境を全部整える |
| 機能 | clustering の出力スキーマ検証テストを追加する | 3 分析のテストを全部作る |
| UI | 審査タブの合成 vs 生の差分強調を改善する | 画面全体を作り直す |
| テスト | association の正常系・境界系テストを追加する | テストを整備する |
| リファクタ | `_per_patient_features` の重複計算を共通化する | コードをきれいにする |

## 分割の指針
- 「〜を全部」「よしなに」「整備する」は粒度過大のサイン → 動詞 + 対象 + 完了条件に割る
- 検証方法を 1 つも書けないタスクは、まず Investigation Issue にする
- 依存があるなら先行 Issue を切り、`depends on #` で明示（暗黙依存を本文に埋め込まない）

## このリポジトリ特有の注意（各 Issue のリスク欄へ）
- 再現性: seed / `RANDOM_STATE=42` を変えない
- 純関数性: analyses.py は外部状態・I/O に依存させない
- 生成物: site/data, site/fragments は手編集せずジェネレータ経由で再生成
- 核メッセージ: 「合成 ≈ 生だが非同一」を壊さない
- フロント: CDN のみ（npm/ビルド工程を足さない）
