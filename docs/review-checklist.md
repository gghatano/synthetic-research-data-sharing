# Review Checklist

レビュー（code-reviewer / security-reviewer）と self-review で使う観点。詳細仕様は
[SPEC.md](SPEC.md) が正。

## 仕様適合性
- [ ] SPEC.md の該当項目を満たしているか（節番号を引用）
- [ ] 仕様にない挙動を勝手に追加していないか
- [ ] 曖昧な仕様について、採用した仮定が明記されているか

## 設計整合性
- [ ] 既存のディレクトリ構成・責務分担（generator=計算/生成, site=表示）に沿うか
- [ ] 既存の命名規則・エラーハンドリング・コメント方針に沿うか
- [ ] 不要な抽象化や過剰設計がないか

## このリポジトリ固有（最重要）
- [ ] `analyses.py` の各関数が**純関数**（データ dict → 結果 dict、外部状態・I/O 非依存）を保つ
- [ ] **再現性**: 乱数はシード固定（generate の seed / `RANDOM_STATE=42`）。固定が崩れていない
- [ ] **生成物を手編集していない**（site/data, site/fragments は make build の出力）
- [ ] **「合成 ≈ 生だが非同一」** の核メッセージを壊していない
- [ ] フロントに **npm / ビルド工程 / バンドラ** を持ち込んでいない（CDN のみ）
- [ ] テンプレ（fragment.html.j2）が**本番 htmx へ移植可能**な形を保つ

## 品質
- [ ] 対応するテストが `tests/` に追加・更新されている
- [ ] `pytest` / `ruff check` / `ruff format --check` / `mypy generator` が通る
- [ ] `make build` が生成物を再生成できる
- [ ] 例外系・境界値（空グループ、欠損、単一観測など）が考慮されている

## セキュリティ / プライバシー
- [ ] Jinja2 autoescape 有効。`| safe` は内部生成の信頼できる値に限定
- [ ] 将来 FastAPI 化を見据え、外部入力（分析名等）を信頼しない形か
- [ ] 生データ隔離の前提を壊す導線・露出がないか（本デモは架空データだが設計として）
- [ ] 秘密情報・実患者データがコード／生成物／コミットに混入していない
- [ ] 追加依存・CDN バージョン固定が適切。CI permissions は最小か

## 保守性
- [ ] 変更範囲が過大でないか（1 PR = 1 タスク）
- [ ] 後続実装の妨げになる暫定実装・TODO 放置がないか
- [ ] コメントは必要最小限で、実装意図（特に臨床的素性・シグナル源）が明確か
