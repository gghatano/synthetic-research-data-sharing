その運用設計なら、かなり良いです。
前回の「仕様書 → 実装計画 → サブエージェント → レビュー」より、さらに一段実務向きにするなら、**GitHub Issueを唯一の作業単位・進捗記録・レビュー証跡にする**形が適しています。

特に、以下の設計がよいです。

```text
仕様書
  ↓
Issue分割
  ↓
Issueごとに branch / worktree 作成
  ↓
Claude Codeが該当Issueを読む
  ↓
サブエージェントが実装・テスト・レビュー
  ↓
PR作成
  ↓
Issueへ進捗・検証結果・残課題をコメント
  ↓
Issue close
```

Claude Code公式ドキュメントでも、worktreeは独立した作業ディレクトリとブランチを持たせ、複数のClaude Codeセッションを分離して進める用途に適すると説明されています。GitHub CLIも `gh issue create`、`gh issue view`、`gh issue develop` などでIssueとブランチの連携を扱えます。([Claude Code][1])

## 追加すべき運用コンセプト

ここで重要なのは、**Issueを単なるToDoではなく、Claude Codeへの作業指示書にする**ことです。

Issueには最低限、以下を入れるべきです。

```text
- 背景
- 対象仕様
- 実装スコープ
- 非スコープ
- 変更想定箇所
- 完了条件
- 検証方法
- 関連Issue
- 既知のリスク
- Claude Codeへの作業指示
```

逆に、Issueに入れるべきでないものは以下です。

```text
- 大きすぎる機能全体
- 未確定の設計論点
- 他Issueに依存する作業を暗黙に含むもの
- 「よしなに改善」系の指示
- テスト方法が定義できないタスク
```

Issue 1件の粒度は、**1つのPRで完結し、レビュー可能で、失敗時に切り戻せる単位**が適切です。

## Issue作成時の粒度ルール

Issueの粒度は、以下のように分けるのが実務上扱いやすいです。

| 粒度    | 適切なIssue例                 | 不適切なIssue例    |
| ----- | ------------------------- | ------------- |
| 調査    | 認証方式の既存実装を確認する            | 認証全体をよしなに作る   |
| 基盤    | APIクライアントの共通エラー処理を追加する    | API層を全部作る     |
| 機能    | ユーザー登録APIを実装する            | ユーザー管理機能を全部作る |
| UI    | ログインフォームを実装する             | 画面全体を完成させる    |
| テスト   | ユーザー登録APIの正常系・異常系テストを追加する | テストを整備する      |
| リファクタ | 重複するバリデーション処理を共通化する       | コードをきれいにする    |

目安は、**Claude Codeが1セッションで理解・実装・検証・説明できる範囲**です。
大きすぎるIssueは、Claude Codeが仕様解釈・実装・テスト・設計判断を同時に抱え、品質が落ちます。

## Issue → worktree → branch の標準コマンド案

GitHub CLIとgit worktreeを使う前提なら、Claude Codeに以下の標準手順を覚えさせるとよいです。`gh issue develop` はIssueに紐づく開発ブランチを管理するコマンドとして提供されています。([GitHub CLI][2])

```bash
# Issue確認
gh issue view 123 --comments

# Issueに紐づくbranchを作成
gh issue develop 123 --name issue/123-user-registration-api --base main

# branch取得
git fetch origin issue/123-user-registration-api

# worktree作成
git worktree add ../worktrees/repo-issue-123 issue/123-user-registration-api

# worktreeへ移動
cd ../worktrees/repo-issue-123

# Claude Code起動
claude
```

Gitの公式ドキュメント上も、`git worktree add -b <new-branch> <path>` により、新しいブランチを作成して別作業ディレクトリへcheckoutできます。([Git][3])

`gh issue develop` を使わない場合は、こちらでもよいです。

```bash
# branchとworktreeを同時作成
git worktree add -b issue/123-user-registration-api ../worktrees/repo-issue-123 main

# GitHubへpush
git push -u origin issue/123-user-registration-api
```

## サブエージェントの役割修正版

Issue駆動にするなら、サブエージェントは以下に再編した方がよいです。

| サブエージェント | 主な役割 | 起動タイミング |

## `issue-runner` サブエージェント定義案

`.claude/agents/issue-runner.md` はこのような内容がよいです。

```markdown
---
name: issue-runner
description: GitHub Issueを読み、スコープに限定して実装・検証・PR準備まで進行するエージェント
tools: Read, Grep, Glob, Bash, Edit, Write
---

あなたはIssue駆動開発の実行担当です。

必ず以下の順序で作業してください。

1. `gh issue view <issue-number> --comments` でIssue本文とコメントを読む
2. Issueの背景、対象仕様、スコープ、非スコープ、完了条件、検証方法を要約する
3. 変更対象ファイルを特定する
4. 実装方針を短く整理する
5. Issueスコープに限定して実装する
6. 必要なテストを追加または更新する
7. 利用可能な検証コマンドを実行する
8. 失敗した場合は原因を分析し、修正して再実行する
9. code-reviewerにレビューを依頼する
10. セキュリティ影響がある場合はsecurity-reviewerにレビューを依頼する
11. 指摘事項を修正する
12. Issueコメント用の進捗サマリとPR本文案を作成する

制約：
- Issueに記載されていない大きな変更をしない
- 非スコープに記載された内容を実装しない
- スコープ外の発見事項は別Issue候補として記録する
- 検証していない内容を完了と主張しない
- テスト失敗を無視しない
- 仕様上の仮定は明記する
````

## `issue-driven-development` SKILL案

```markdown
---
name: issue-driven-development
description: GitHub Issueを作業単位として、branch/worktree/PR/検証記録を一貫管理する
---

# Issue Driven Development

このリポジトリでは、すべての実装作業をGitHub Issue単位で管理する。

## 原則

- 1 Issue = 1 branch = 1 worktree = 原則1 PR
- Issue本文を作業指示書として扱う
- Issue本文にない作業を勝手に実装しない
- スコープ外の発見事項は別Issue候補として記録する
- 作業完了時はIssueとPRに検証結果を残す

## 作業開始

1. Issue番号を確認する
2. `gh issue view <issue-number> --comments` を実行する
3. 背景、対象仕様、スコープ、非スコープ、完了条件、検証方法を読み取る
4. 現在のbranchとworktreeがIssue専用であることを確認する
5. 実装方針を短く整理する

## 実装

1. 関連ファイルを確認する
2. 既存パターンに合わせて実装する
3. 必要なテストを追加・更新する
4. 仕様にない拡張は行わない

## 検証

1. format
2. lint
3. typecheck
4. test
5. build

利用可能なものを実行し、実行できないものは理由を記録する。

## レビュー

1. code-reviewerでレビューする
2. セキュリティ影響があればsecurity-reviewerでレビューする
3. 指摘事項を修正する
4. 再検証する

## 完了

1. PR本文を作成する
2. Issueへ進捗コメントを残す
3. PRとIssueを相互リンクする
4. 未対応事項を明記する
```

## Issue分割の実務ルール

最初から全Issueを作り切るより、以下の三層に分けるのがよいです。

```text
Epic Issue
  └─ Feature Issue
       └─ Implementation Task Issue
```

ただし、Claude Codeに解かせるのは最下層だけです。

### Epic Issue

```text
例：
[EPIC] ユーザー認証機能
```

目的、全体スコープ、関連仕様、完了条件、配下Issue一覧を記載。

### Feature Issue

```text
例：
[Feature] メールアドレス・パスワードによるログイン
```

機能単位。設計論点や依存関係を整理。

### Implementation Task Issue

```text
例：
[Task] ログインAPIのリクエストバリデーションを実装する
```

Claude Codeが実装する単位。

この分け方にすると、計画管理はGitHub Issueで見える一方、Claude Codeに渡すIssueは小さく保てます。

---

## 運用上の強い推奨

設計としては、**Issue本文をClaude Codeへのプロンプトそのものにする**のが最も安定します。
つまり、毎回長いプロンプトを書くのではなく、Issue側に必要なコンテキストを集約し、Claude Codeにはこう言えばよい状態にします。

```text
Issue #123 を読んで、このworktreeで解決してください。
```

そのためには、Issueテンプレートを強めにする必要があります。

特に重要なのは以下です。

```text
非スコープ
完了条件
検証方法
```

この3つが弱いIssueは、Claude Codeが過剰実装・周辺修正・検証不足に寄りやすくなります。

---

## 最終的な推奨フロー

```text
1. 仕様書を読む
2. spec-analystで実装候補を抽出
3. issue-plannerでIssue候補に分割
4. 人間がIssue候補を確認
5. Claude Codeがgh issue createでIssueを作成
6. Issueごとにbranch/worktree作成
7. worktree内でClaude Code起動
8. Issue本文とコメントを読む
9. implementation-agentが実装
10. test-engineerが検証
11. code-reviewerがレビュー
12. security-reviewerが必要時レビュー
13. PR作成
14. Issueへ検証結果をコメント
15. merge後Issue close
16. worktree削除
```

この形にすると、Claude Codeの作業が「会話ログ」ではなく、**GitHub Issue / PR / commit / CI結果に残る開発プロセス**になります。
チーム開発・後続レビュー・引き継ぎを考えるなら、この構成が最も扱いやすいです。
