# 仕様書: synthetic-research-data-sharing

## 1. 背景・目的

医療・研究の現場では、患者データのように機微で再配布できないデータを外部の分析者と
共有したい場面がある。生データをそのまま渡すことはプライバシー・規制の観点で困難な
ため、本デモは **code-to-data / TRE（Trusted Research Environment）型** のワークフローを
提示する。

- 生データは TRE 内に隔離し、外部に出さない
- 分析者にはデータではなく **合成データ** を渡し、分析コードを開発してもらう
- 開発されたコードを **オーナーが生データに適用** して本物のレポートを得る

**核となるメッセージ**: 同一の分析コードが合成データ・生データの両方で動作し、結果は
**統計的に近いが完全には一致しない**。このギャップが「合成データで安全に開発し、最後に
生データへ適用する」というワークフローの価値を示す。

> **位置づけ（#54）**: この「合成 vs 生の核メッセージ」は、UI 上では**合成 vs 生を並置する解説面
> （原理デモ面）を設けず**、実コンテンツ（カタログ閲覧・合成データのダウンロード・提出・審査）の
> 体験そのものでワークフローを示す。カタログ詳細・提出・審査の各ビューには合成 vs 生の並置を出さない。
> 合成データで開発したコードを生データへ適用する原理は、提出物（プログラム＋結果＋レポート）を
> オーナーが審査・承認する流れの中で説明する。

## 2. 用語

| 用語 | 説明 |
|------|------|
| TRE | Trusted Research Environment。生データを隔離し、計算だけを内部で行う環境 |
| code-to-data | データを動かさず、分析コードをデータ側へ持ち込んで実行する方式 |
| 合成データ | 生データの統計的性質を模して生成した架空データ。個票は実在しない |
| nadir | 治療後に PSA が到達する最低値 |
| Phoenix 基準 | 生化学的再発の定義。PSA ≥ nadir + 2.0 ng/mL を進行イベントとする |

## 3. データモデル

各データセットは 3 テーブル（JSON）。カタログ型対応（#22）で複数データセットを扱う。

### 3.0 カタログとパス規約（複数データセット, #22）

generator は **複数データセット**を `DATASETS` レジストリ（`generator/generate_data.py`）で
定義し、以下を生成する。

- 各データセットの生/合成データ: `site/data/<dataset_id>/{raw,synthetic}.json`
- 各データセットの分析フラグメント: `site/fragments/<dataset_id>/{analyst,owner}/<analysis>.html`
  （`analyst` = 合成データ結果、`owner` = 生データ結果）
- カタログ索引: `site/data/catalog.json`（全データセットのメタデータ一覧）

**カタログ索引 `catalog.json`** の形:

```json
{ "datasets": [ { /* メタデータ */ }, ... ] }
```

各エントリの**メタデータスキーマ**（必須キー）:

| キー | 型 | 説明 |
|------|----|------|
| `dataset_id` | str | 一意なデータセット ID（パスにも使う、例 `prostate-psa`） |
| `title` | str | 表示名 |
| `description` | str | 概要 |
| `owner` | str | データオーナ（擬似アカウント） |
| `domain` | str | ドメイン（例 `oncology/urology`） |
| `tags` | list[str] | 検索/分類用タグ |
| `usage_examples` | list[str] | 想定分析の短い活用例 |
| `n_patients` | int | 合成データの患者数 |
| `paths` | object | `raw`/`synthetic`/`fragments_analyst`/`fragments_owner` の参照パス |
| `dummy_preview` | object | カタログ表示用の少量サンプル（各テーブル先頭数行、**合成データ由来**） |

> `dummy_preview` は公開可能な**合成データ**から抽出する（生データは出さない）。

### 3.1 テーブル（3 テーブル）

`site/data/<dataset_id>/{raw,synthetic}.json`。

### patients
| 列 | 型 | 説明 |
|----|----|------|
| `patient_id` | str | 患者ID（例: `RAW-0001` / `SYN-0001`） |
| `age` | int | 年齢 |
| `risk_group` | enum | `low` / `intermediate` / `high` |
| `enrollment_date` | date | 登録日 |

### psa_measurements
| 列 | 型 | 説明 |
|----|----|------|
| `patient_id` | str | 患者ID |
| `date` | date | 測定日（約3ヶ月毎、最大13回 ≒ 36ヶ月） |
| `psa` | float | PSA 値 (ng/mL) |

### medications
| 列 | 型 | 説明 |
|----|----|------|
| `patient_id` | str | 患者ID |
| `datetime` | datetime | 投薬日時（ADT 開始） |
| `drug` | enum | `leuprolide` / `goserelin` / `bicalutamide` |
| `dose_mg` | float | 用量 (mg) |

### 生成の素性（`generator/generate_data.py`）

- ADT（アンドロゲン除去療法）開始後、PSA は指数的に nadir まで低下する
- 用量が多いほど nadir が深くなる（= 投薬–反応の関連分析のシグナル源）
- リスク群に応じた確率で nadir 後に PSA が再上昇 → 進行イベント
- リスク群が高いほど baseline PSA が高く、進行確率・再上昇速度が大きい

### 生データ vs 合成データ

合成データは **別シード + 分布シフト**（baseline をやや上げ、進行確率・再上昇速度を
やや上げる）で生成する。両者は統計的に類似するが、個々の値・クラスタ境界・KM 曲線は
完全には一致しない。これにより核メッセージが成立する。

> 本デモのデータは **すべて架空**であり、実在の患者に由来しない。

## 4. ワークフロー（カタログ連携の提出・審査フロー）

ライブ UI はカタログ型である。`site/index.html` の Alpine.js が、登録済みデータセットの
カタログと、データセットに紐づく提出物（submissions）を in-memory で管理する。中心となる
フローは以下のとおりで、ブラウザ内ワークベンチや分析フラグメントの htmx 読込は**存在しない**。

```
 register()              download()            submit()              approve()
オーナーが ──────► 分析者が合成データを ──────► 分析者がプログラム＋ ──────► オーナーが提出物を
合成データを        ダウンロード              結果＋レポートを提出       審査して承認
登録                （JSON / CSV）            （status: submitted）     （status: approved）
```

| フェーズ | ロール | 動作 |
|----------|--------|------|
| 登録 | オーナー：合成データ登録 | 合成データセットをカタログに登録（生データは🔒非公開のまま）。詳細は **§7.7** |
| ダウンロード | 分析者：取得 | 個別ページから合成データを JSON / CSV でダウンロードし、手元で分析プログラムを開発・実行 |
| 提出 | 分析者：提出 | プログラム＋結果＋レポートをアップロードして提出。提出物は `status: "submitted"` でデータ個別ページの「紐づく提出物」一覧に紐づく。詳細は **§7.8** |
| 審査 | オーナー：審査・承認 | 個別ページの提出物カードから選び、アップロードされたプログラム・結果・レポートを閲覧して承認。`status` を **`submitted → approved`** に遷移。詳細は **§7.9** |

提出物は共有ストア `Alpine.store('submissions')`（in-memory）で保持し、承認は同ストア上で
更新されて一覧へ即時反映される。owner と非 owner で審査導線の出し分け（擬似ガード）を行う
（合成 vs 生の並置は行わない）。

> **歴史的経緯（旧 3 ロールデモ, #54 で撤去）**: 初期デモは Alpine.js が
> `idle → published → submitted → approved` の 4 状態を持ち、分析者が画面内ワークベンチで
> 分析を選択して `fragments/<id>/analyst/<a>.html` を htmx で読み込む形だった。この
> `published` 状態と画面内ワークベンチ／分析フラグメントの htmx 読込は #54 で撤去済みで、
> **現行 UI には存在しない**。本節は概念的背景としてのみ記し、現行の機構は上記
> カタログ連携フローである。

## 5. 3 つの分析（`generator/analyses.py`）

各分析は「データセット dict → 結果 dict」の **純関数** として実装し、raw・synthetic の
双方に同一コードを適用する。

### 5.1 PSA軌跡クラスタリング `clustering`
- 患者ごとに特徴量を抽出: `baseline`, `nadir`, `time_to_nadir`, `late_slope`（nadir後の傾き）, `variability`
- 標準化後 `sklearn.cluster.KMeans(k=3)` で分類
- `late_slope` の平均で responder / partial / progressor を解釈付け
- 出力: クラスタ別人数・標準化重心・クラスタ別平均 PSA 軌跡

### 5.2 投薬–PSA反応の関連分析 `association`
- 患者ごとの PSA 低下率 `100*(baseline-nadir)/baseline` を算出
- 薬剤別サマリ（平均低下率・SD・平均用量）
- 用量 vs 低下率 を `numpy.polyfit` で単回帰（傾き・切片・R²）
- 出力: 薬剤別テーブル・回帰直線・散布図サンプル点

### 5.3 進行イベントの生存時間分析 `survival`
- Phoenix 基準（PSA ≥ 累積nadir + 2.0）で各患者の進行時刻 or 打ち切りを判定
- リスク群別に Kaplan-Meier 推定（手実装の階段曲線）
- 出力: リスク群別 KM 曲線・イベント率・進行までの中央値

## 6. モック実装（htmx + Alpine + Python生成、GitHub Pages）

- フロント: 素の HTML + htmx + Alpine.js + Chart.js（すべて CDN、ビルド工程なし）
- 分析・データ: Python（pandas / numpy / scikit-learn）で生・合成データを生成し、3分析を
  両データに事前計算。結果を Chart.js 設定 JSON ＋ Jinja2 でレンダリングした HTML
  フラグメントとして出力
- 状態管理: Alpine.js の in-memory ストアでカタログ（`Alpine.store('catalog')`）と
  提出物（`Alpine.store('submissions')`）を保持。提出物の `status` は **`submitted → approved`**
  に遷移する（`published` 状態は存在しない）。ライブシェルは分析フラグメントを htmx で
  読み込まない（画面内ワークベンチは #54 で撤去済み）
- セキュリティ上の注意: モックでは `site/data/<id>/raw.json` および owner（生データ）
  フラグメントも GitHub Pages から実際に配信される（データは架空ゆえ許容）。本番では
  生データ実体・生データ結果・`catalog.json` の `paths.raw` を公開成果物に含めず TRE 内に
  隔離する。

## 7. 本番想定アーキテクチャ（FastAPI + htmx）との対応

| 関心事 | モック（本リポジトリ） | 本番プロトタイプ |
|--------|------------------------|------------------|
| フロント | htmx + Alpine + Chart.js（CDN） | 同一 |
| フラグメント | 事前生成した静的 `.html` | FastAPI が Jinja2 で同形 HTML を返す（`generator/templates/` を流用） |
| 分析実行 | ビルド時に事前計算 | リクエスト時にサーバーで実行（`generator/analyses.py` を流用） |
| 認可・隔離 | 生データを「非公開」表示で擬似 | TRE 内に隔離、分析者は生データへアクセス不可 |
| ワークフロー | Alpine のクライアント状態 | サーバー側ジョブ状態 + DB |

移植時に置き換わるのは **「事前生成 → サーバー応答」** の供給経路のみ。htmx マークアップ
（`generator/templates/fragment.html.j2`）と分析コード（`generator/analyses.py`）はそのまま
再利用できる。

## 7.5 擬似認証・画面シェル・ロール（owner / analyst）

カタログ型プラットフォームへの移行に向けた **最初の画面シェル**。ログインを起点に、
ロールに応じた導線を出し分ける。バックエンドを持たないため認証・状態保持はすべて
**擬似（クライアント側 in-memory）** で行う。

### ロール

| ロール | 識別子 | 想定操作 |
|--------|--------|----------|
| データオーナー | `owner` | 合成データの登録（データ登録）／データ探索 |
| 分析者 | `analyst` | データ探索（合成データで分析を開発・提出） |

### 擬似認証と状態保持

- ログイン画面は **ユーザー名＋パスワード（擬似）** で行う（#66 OPTION A）。
  **実際の認証・ハッシュ化は一切しない**（メモリ上で平文を照合するだけのモック）。
  **ロールは画面で選ばない**：入力ユーザー名に一致した**プリセットアカウント**の `role` を採用する。
  - **プリセットアカウント**（デモ）は `role` と**デモ用パスワード（平文定数）**を持つ。
    例: データオーナー（`owner` / `owner-demo`）、分析者（`analyst` / `analyst-demo`）。
    プリセットのチップを選ぶとユーザー名とデモ用パスワードが**両方**自動入力される。
    モックのため、ログインできるよう**デモ用パスワードは画面のヒント（`login-hint`）でも開示**する。
  - `login(name, password)`：入力ユーザー名に一致するプリセットを探し、パスワードも一致したら
    `user = { name, role }`（role はアカウント由来）を設定して `top` へ。一致しなければ
    エラー（「ユーザー名またはパスワードが違います」）を `x-text`（XSS 安全）で表示する。
    **パスワードは `user` に保持しない**（照合のみ）。
- ログイン状態は **Alpine の共有ストア `Alpine.store('session')`（in-memory）** に保持する。
  - 単一ページのビュー切替（SPA 的: `login → top → register / explore`）で、ビュー遷移を
    またいでログイン状態・ロールを保持する。
  - **`localStorage` は使わない。** ハードリロードで状態はリセットされる（既存 3 ロール
    デモと同じ思想）。これはデモ用途として許容する割り切り。
- ログアウトでストアを破棄し、ログイン画面へ戻る。
- **カタログ／個別ページはログイン不要**で閲覧でき、ログイン画面の「ログインせずカタログを見る」
  からも到達できる（公開閲覧。詳細は §7.6）。**owner 専用ガード**（`register` ビュー、提出物の
  審査・承認）は従来どおり保持し、**アカウント由来の `user.role` で判定**する。

### 画面シェルと擬似ガード

- ログイン後トップ（`top`）から **「データ登録」「データ探索」** へ遷移する。
  - **データ登録**は `owner` のみ可視（owner 専用ビュー）。中身は後続タスク（PF-4）で実装。
  - **データ探索**は全ロール可視。当面は既存の 3 ロールデモ（公開/分析/審査）を内包し、
    後続タスクでカタログ型 UI（PF-3）へ段階移設する。
- **擬似ガード**: 未ログインで保護ビュー（top / register / explore）へ遷移しようとしても
  ログイン画面に留める。`analyst` が owner 専用ビュー（register）へ来たらトップへ戻す。

### 本番との対応（重要）

ここでのログイン・ロールガードは **あくまで UI 上の擬似**であり、セキュリティ境界ではない。
本番では **TRE/サーバー側のセッションと認可で隔離**し、生データやエンドポイントへのアクセス
可否はサーバーが判定する。クライアント側のロール出し分けは UX のためのものに過ぎない。

| 関心事 | モック（本リポジトリ） | 本番プロトタイプ |
|--------|------------------------|------------------|
| 認証 | ユーザー名＋パスワード（擬似・平文照合。ロールはアカウント由来） | IdP / サーバーセッション |
| 状態保持 | Alpine 共有ストア（in-memory） | サーバーセッション + DB |
| 認可・ロールガード | クライアント側の出し分け（UX 用） | サーバー側で強制（TRE 隔離） |

## 7.6 データカタログ閲覧（一覧・個別ページ, #23）

カタログ型プラットフォームの**閲覧（読み取り）機能**。誰でも研究用データの
**メタデータ・ダミーデータ・活用例**を確認できる。供給源は PF-2（#22）が事前生成する
`site/data/catalog.json`。ビューは `site/index.html` の Alpine ストア
`Alpine.store('session').view` に追加した `explore`（一覧）/`dataset`（個別）で切り替える。

### 公開閲覧（ログイン不要）

- カタログ一覧（`explore`）・個別ページ（`dataset`）は**誰でも閲覧可**。擬似ガード
  （`isPublicView`）の対象外とし、未ログインでもアプリバーの「ログイン」やログイン画面の
  「ログインせずカタログを見る」から到達できる。
- 保護ビュー（`top` / `register`）は従来どおり擬似ガードの対象（未ログインはログインへ、
  `analyst` の `register` 到達はトップへ戻す）。

### カタログ一覧（`explore`）

- `data/catalog.json` を `fetch` し、`datasets[]` を**カード**で描画
  （title / owner / domain / n_patients / description / tags）。
- 簡易検索（最小実装）: タイトル・概要・オーナー・ドメイン・タグの部分一致で絞り込む。
  件数表示（`<絞込数> / <総数> 件`）を添える。一致 0 件は空状態を表示。
- 各カードの「詳細を見る」で個別ページへ（`openDataset(dataset_id)`）。

### データ個別ページ（`dataset`）

選択中の `currentDatasetId` に対応する `catalog.json` のエントリを描画する。

- **メタデータ**: title / owner / domain / n_patients / description / tags。
- **活用例**: `usage_examples` を箇条書き表示。
- **ダミーデータプレビュー**: `dummy_preview` の各テーブル（patients / psa_measurements /
  medications）を列順固定のテーブルで表示。これは**合成データ由来**であり生データは含まない。
- **合成データのダウンロード（#40, N1）**: 分析者が合成データを手元に持ち帰り、外部で分析
  プログラムを開発できるよう、個別ページに**合成データ（JSON）のダウンロード導線**を置く。
  - 配布対象は事前生成済みの `data/<dataset_id>/synthetic.json`（架空の合成データ）のみ。
    `<a download>` で取得し、ファイル名は `<dataset_id>-synthetic.json`。
  - **生データ（`raw.json`）はダウンロードさせない**（TRE 隔離の擬似を維持。UI から一切
    リンクしない）。「これは合成データであり生データではない」旨を注記する。
  - **仮定**: 合成データファイルを持つのは**同梱データセット（base）のみ**。ユーザー登録分
    （`user-<n>`）は合成データ未生成のため、ダウンロード導線を出さない（§7.7 の「分析未提供」
    と同じ扱い）。本番では TRE が登録時に合成データを用意する。
  - **CSV ダウンロード（#51）**: JSON に加え、**テーブル単位の CSV** も提供する。クリック時に
    `synthetic.json` をブラウザ内で fetch し、`meta` を除く top-level の各配列テーブル
    （`patients` / `*_measurements` / `medications` …）を **RFC4180 準拠**の CSV へ変換して
    Blob ダウンロードする（クライアント側変換のため `generator/` も `site/data/**` も変更しない）。
    ファイル名は `<dataset_id>-<table>-synthetic.csv`。ヘッダは先頭行のキー順、カンマ・二重引用符・
    改行を含むセルは引用符で囲み内部引用符を二重化。データは標準 DOM API のみで扱い
    `innerHTML`/`x-html` 注入はしない。**生データ（raw）は CSV でもダウンロードさせない**。
    CSV 導線も同梱データセット（base）のみで表示し、ユーザー登録分には出さない。
- **分析プログラム＋結果の提出（#41, N2）**: ブラウザ内の「分析ワークベンチ」（3 分析を
  htmx で実行）は**廃止**した。実際の code-to-data に合わせ、個別ページには**プログラム＋結果
  アップロード提出フォーム**を置く（合成データを手元に持ち帰り外部で開発 → 提出）。詳細は
  §7.8。同梱データセットのみ提出可。ユーザー登録分（`user-<n>`）は合成データ未生成のため
  提出フォームを出さず注記のみ（`dataset-no-submission`、§7.7 と同扱い）。
  - **「合成 vs 生（合成≈生）の核メッセージ」を並置する解説面は UI に設けない**（#54）。原理は
    合成データのダウンロード→提出→審査という体験の流れの中で示す。
- **紐づく提出物の枠**: 下記の共有ストアから `dataset_id` で引いた提出物を一覧表示する
  （プログラム名・結果件数・レポート・status の概要）。提出物の詳細閲覧と審査は §7.9（#42）。

### 提出物の共有ストア（枠のみ。#25/#26 への引き継ぎ）

提出物は **Alpine 共有ストア `Alpine.store('submissions')`（in-memory）** で保持する。
**キー設計は `dataset_id` 起点**（`catalog.json` / パス規約と一致）。

```
Alpine.store('submissions') = {
  byDataset: { [dataset_id]: Submission[] },  // dataset_id を起点キーにする
  forDataset(dataset_id) -> Submission[],      // 読み取り(個別ページが使用)
  add(dataset_id, submission),                 // #41 の提出が書き込む
  find(dataset_id, submission_id) -> Submission|null,
  approve(dataset_id, submission_id, reviewer) // #42 の審査が status を遷移
}
```

- 個別ページの「紐づく提出物」は `forDataset(dataset_id)` を読むだけ。
- 提出（#41）は `add()` 経由で `byDataset[dataset_id]` に積む。`Submission` スキーマは
  **§7.8 で確定**（プログラム＋結果アップロード型: `{ id, datasetId, program{name,text},
  results[{type,name,content}], report, analyst, status, submittedAt }`）。審査（#42）は
  `approve()` で `status` を遷移させる。
- `localStorage` 不使用（ハードリロードでリセット = 既存デモと同じ思想）。

### 本番との対応

| 関心事 | モック（本リポジトリ） | 本番プロトタイプ |
|--------|------------------------|------------------|
| カタログ供給 | 事前生成 `catalog.json` を `fetch` | カタログ API（DB 検索） |
| ダミーデータ | `dummy_preview`（合成データ抽出） | 合成データのサンプル応答 |
| 分析フラグメント | `htmx.ajax` で静的 `.html` | FastAPI が同形 HTML を返す |
| 提出物 | Alpine 共有ストア（in-memory） | サーバー側ジョブ状態 + DB |

## 7.7 データ登録（擬似, #24）

カタログ型プラットフォームの**登録（書き込み）機能**（機能1）。データオーナーが研究用データの
**メタデータ＋合成/ダミーデータ**をカタログに新規登録できる。ビューは `register`（owner 専用）。

### 確定仮定（静的モックでの登録の表現）

- **ブラウザ上で合成データ生成は行わない**（Python を動かせないため）。登録は
  **メタデータ＋（事前同梱サンプルから選択 or CSV アップロードした）ダミーデータプレビュー**を
  共有ストアに追加する**擬似**とする。
- 本番では **TRE（Trusted Research Environment）内で生データから合成データを生成・検証**し、
  その結果（メタデータ＋ダミー／合成）をカタログ API に登録する。モックはこの UX のみを再現する。
- 登録分は **in-memory（`Alpine.store('catalog')`）**。`localStorage` 不使用でハードリロードで消える
  （既存デモと同じ思想）。**扱うデータはすべて架空**（フォームにも明記）。

### デモ用フォーム記入（便宜機能, #55 → #62 で挙動変更）

- `register` ビュー（owner 専用）に **「デモ用データでフォームを記入」ボタン**を備える。手入力なしで
  **同梱の架空デモデータ**で下の手入力フォームを**埋めるだけ**で、登録もカタログへの反映も
  個別ページへの遷移も**しない**（#62）。ダミーデータ指定も `source=sample` ＋ **有効な同梱サンプル**
  をセットして submit-ready にするため、ユーザーが「カタログに登録する」(`reg-submit`) を押せば
  通常の手入力と同じ経路で登録される。登録後もカタログへ自動遷移はせず `register` ビューに留まる
  （ナビゲーションはユーザーに委ねる）。
- 登録は手入力フォームと**同一の `catalog.register()` 経路**を通す（`reg-submit` 経由のみ）ため、
  登録結果は手動登録と同形・同挙動。デモデータ定義はページ内 JS 定数（複数件をクリックごとに巡回し、
  異なる定義でフォームを埋める）。**アーキテクチャは不変**で、追加のガード・ストア・on-disk ファイルは
  作らない（owner 専用 `register` ビュー内に閉じる）。
- こうして登録されたデモデータも `user-<n>` 採番のユーザー登録分であり、`data/<id>/synthetic.json` を**持たない**ため、
  合成データ JSON/CSV ダウンロード（#40 / #51）は**対象外**（`hasSyntheticDownload=false`）。
  個別ページは到達可・提出フローも動くが、合成データ DL 導線は出ない（仕様どおり）。

### owner 専用ガード

- ナビ「データ登録」とトップの登録カードは `owner` のみ表示（#21 の出し分け）。
- `register` ビュー自体も擬似ガード（`go('register')` は `owner` 以外をトップ/ログインへ戻す）。

### 入力項目

- **メタデータ**: title（必須）/ description（必須）/ domain（必須）/ n_patients / tags（カンマ区切り）/
  usage_examples（活用例、1 行 1 件）。
- **ダミーデータの指定**（必須・いずれか）:
  1. **事前同梱サンプル**から選択（ブラウザ内蔵の固定ダミー。生成ではない）。
  2. **CSV アップロード**（`FileReader.readAsText` → クライアント側で **RFC4180** 準拠にパース、#63）。
     **1 ファイル = 1 テーブル**。先頭行がヘッダ（列名）の CSV で、ヘッダの列名を
     **許可テーブル/列の whitelist**（`patients` / `psa_measurements` / `medications`）と突き合わせて
     テーブルを自動判定し、許可列のみ・先頭数行を `dummy_preview` に整形する。クォート内の
     `,` `"` 改行・`""` エスケープ・先頭 BOM・末尾空行を扱う。**合成データの CSV ダウンロード（#51）
     と同形**。**壊れた CSV・未知ヘッダでも UI を破綻させない**（状態メッセージで通知し登録を止める）。

### カタログ共有ストア（基底 + ユーザー登録分のマージ）

カタログの単一ソースを **`Alpine.store('catalog')`** に集約する。一覧（`explore`）・個別ページ
（`dataset`）はこのストアの **`all()`** を唯一の表示元として読む（#23 の直接 `fetch` から切替）。

```
Alpine.store('catalog') = {
  base: Dataset[],                 // 事前生成 catalog.json 由来(読み取り専用の基底)
  user: Dataset[],                 // owner が登録した擬似分(in-memory)
  ensureLoaded(): Promise          // base を一度だけ fetch(複数ビューから二重 fetch しない)
  all() -> [...user, ...base]      // 一覧/個別が読む唯一のソース(登録分を先頭に)
  find(dataset_id) -> Dataset|null
  isUserDataset(dataset_id) -> bool
  register(dataset) -> dataset_id  // "user-<n>" を採番して登録、即時に一覧へ反映
  remove(dataset_id) / reset()     // 登録分の削除・全リセット(base は不変)
}
```

- ユーザー登録分の `dataset_id` は **`user-<n>`** でユニーク採番（base の id と衝突しない）。
- 登録は **即時にカタログ一覧・個別ページへ反映**される（`all()` を共有しているため）。
- 登録済み一覧（`register` ビュー）に**削除/リセット導線**を備える。

### ユーザー登録分の個別ページ

- 個別ページはユーザー登録データセットでも開ける（メタデータ・活用例・ダミープレビューを表示）。
- ただし**分析ワークベンチ（フラグメント）はユーザー登録分には存在しない**。この場合は
  「登録メタデータ＋ダミーデータのみ（分析フラグメント未生成）」の旨を表示し、ワークベンチを出さない。
  既存同梱データセット（`prostate-psa` / `renal-marker`）の分析は従来どおり動く。

### 本番との対応

| 関心事 | モック（本リポジトリ） | 本番プロトタイプ |
|--------|------------------------|------------------|
| 合成データ生成 | 行わない（同梱サンプル/アップロードで代替） | TRE 内で生成・評価 |
| 登録の永続化 | Alpine 共有ストア（in-memory） | カタログ API + DB |
| 認可 | クライアント側の owner 出し分け（UX 用） | サーバー側で強制 |

## 7.8 合成データの分析プログラム＋結果のアップロード提出（カタログ連携, #41）

カタログ型プラットフォームの**活用（提出）機能**（機能3）。非オーナ（analyst）が公開された
**合成データを手元に持ち帰り、外部で開発した分析プログラムと結果をアップロードして提出**し、
その提出物をデータ個別ページへ紐づける。

> **変更（#41, N2）**: ブラウザ内の「分析ワークベンチ」（3 分析を htmx で実行）は**廃止**した。
> 実際の code-to-data（合成データを外部に持ち出し開発 → 提出 → TRE 内で生データへ適用）に整合
> させ、提出を「プログラム＋結果アップロード」型へ変更。アップロードしたプログラムは**一切実行
> しない**（表示・保管のみ）。旧スキーマ（`analysis`/`analysisTitle`/`codeExcerpt`）から移行。

### 提出フォーム（プログラム＋結果）

- 個別ページの提出フォーム（`submit-form`）で、以下をアップロード／入力する。
  - **分析プログラム（必須）**: `.py` 等のファイルを `FileReader.readAsText` でテキスト取得し、
    `details.frag-code > code` に `x-text` で表示する（**実行はしない**）。サイズ上限 512KB。
  - **結果（画像, 任意）**: PNG/JPG 等を `FileReader.readAsDataURL` で読み、`<img :src>` で
    プレビューする（dataURL のみ保持。サーバー送信なし）。サイズ上限 2MB（dataURL 肥大化防止）。
  - **結果（テキスト/表, 任意）**: textarea。
  - **レポート/メモ（任意）**: textarea。
  - **結果は画像かテキストの少なくとも一方が必須**（両対応。最小は片方）。
- **提出は要ログイン**。未ログイン時は提出ボタンを無効化しログイン導線（`submit-need-login`）を
  出す（**仮定**: 閲覧は誰でも可・提出のみ要ログイン。主体は analyst だが owner も提出可）。
- 入力バリデーション（ファイル種別・サイズの最低限）。壊れた入力・過大サイズは状態メッセージで
  通知し、UI を破綻させない。画像 MIME は PNG/JPEG/WebP/GIF の完全一致（`image/svg+xml` は
  スクリプト混入を避けるため不可）。結果テキスト・レポートは上限なし（デモ前提の in-memory）。
- 提出物は `Alpine.store('submissions').add(dataset_id, {...})` で保存し、個別ページの
  **「紐づく提出物」**へ即時追加する（プログラム名・結果件数・レポート・status の概要を表示。
  詳細閲覧・審査は §7.9, #42）。

#### ワンクリック・デモ提出（便宜機能, #52）

- 提出フォームに **「デモ用のサンプルを提出」ボタン**（`submit-demo`）を備える。ファイルを
  用意せず、ブラウザ内蔵の架空サンプル（分析プログラム＋結果テキスト＋所見）を**手動提出と
  同じ `submissions.add()` 経路**で 1 件保存し、提出→審査フローを素早く試せる（#55 のデモ登録と
  同思想）。サンプルは `DEMO_SUBMISSIONS`（名前付き JS 定数, #53 で再利用可）に複数定義し、
  **クリックごとに次を巡回**して提出する。
- **同じ Submission スキーマ・同じ認証ガード**（要ログイン。未ログイン時はボタン無効化）。
  `analyst` はクリック時のログインユーザー名、`status` は store が `"submitted"` を補完。
  結果は `type:"text"` のみ（大きな dataURL を避ける）。プログラムは**一切実行しない**（表示・
  保管のみ）。サンプル定義は架空で、実患者データ・実行コードを含まない。

#### 仮想（プリセット）提出物の初期配置（便宜機能, #53）

- 提出物一覧／審査が初回から空に見えないよう、起動時（`alpine:init`）に **デモ用の仮想
  （プリセット）提出物**を `submissions.seed()` で 1 度だけ流し込む（冪等。in-memory のため
  ハードリロードでリセット、localStorage 不使用）。提出→審査をすぐデモできる。
- 素材（プログラム＋結果＋レポート）は `DEMO_SUBMISSIONS` を**再利用**し、preset ごとに
  `analyst` 名・`submittedAt`・分析の焦点（report 補足）を差し替える。`status` は **submitted と
  approved を混在**させ（approved は `reviewedBy`/`reviewedAt` を持つ）、一覧・詳細で状態差が見える。
- id は `"preset-<n>"` で明示し、ユーザー提出の `"sub-<n>"` 採番（`seq`）と**衝突しない**
  （`seq` はユーザー提出のみ進む）。UI に小さな「デモ」バッジを付け、**架空の成果物**であることを
  示す。プログラムは一切実行せず、テキスト／ファイル名はすべて `x-text` で表示する。

### 確定 Submission スキーマ（#42 の審査が読む前提）

```
Submission = {
  id:          string,   // "sub-<seq>"（store が採番）
  datasetId:   string,   // 対象データセット（= catalog.json の dataset_id）
  program: {             // アップロードした分析プログラム（実行しない。表示/保管のみ）
    name:      string,   //   ファイル名
    text:      string,   //   ソース文字列（readAsText）
  },
  results: [             // 結果（画像/テキスト。最低 1 件）
    { type: "image", name: string, content: string /* dataURL */ } |
    { type: "text",  name: string, content: string }
  ],
  report:      string,   // 任意のレポート/メモ
  analyst:     string,   // 提出者（ログイン中ユーザー名）
  status:      string,   // "submitted"（初期値。審査で approved へ遷移 = #42）
  submittedAt: string,   // ISO8601
  // 審査（#42）で reviewedBy / reviewedAt を追記する。
}
```

`store.add()` は `id` / `status` / `submittedAt` を未指定なら補完し、保存済みオブジェクトを返す。
**`status` は #42（審査）が後から書き換える**前提で、ここでは常に `"submitted"` を初期値とする。

### セキュリティ（アップロード・表示）

- アップロードしたプログラムテキスト・結果テキスト・ファイル名は必ず `x-text`（エスケープ表示）。
  `x-html`/`innerHTML` は使わない。画像は **dataURL のみ**でプレビューする（任意 URL を踏ませない）。
- アップロードされたプログラムは**一切実行しない**（表示・保管のみ）。

### ユーザー登録データセットの扱い（仮定）

- **仮定**: 合成データを持つ**同梱データセット（`prostate-psa` / `renal-marker`）のみ**提出可。
  ユーザー登録分（`user-<n>`）は合成データ未生成のため提出フォームを出さず注記のみ
  （`dataset-no-submission`、§7.7 の「分析未提供」と同扱い）。本番では TRE が登録時に合成
  データを用意する。

### 本番との対応

| 関心事 | モック（本リポジトリ） | 本番プロトタイプ |
|--------|------------------------|------------------|
| 合成データ取得 | 個別ページから `synthetic.json` をダウンロード（#40） | 合成データ API / TRE からの払い出し |
| プログラム | アップロードしたファイルを表示・保管（実行しない） | TRE 内で隔離実行（サンドボックス） |
| 結果 | 画像（dataURL）／テキストをアップロード | ジョブ生成物（画像・表）をサーバー保存 |
| 提出物保存 | Alpine 共有ストア（in-memory） | サーバー側ジョブ状態 + DB |
| 認可 | 提出のみ要ログイン（UX 用ガード） | サーバー側で強制（ロール・データセット権限） |

## 7.9 オーナーによる審査（提出物の閲覧・承認, #42）

カタログ型プラットフォームの**審査機能**（機能4）。owner が提出物（#41 でアップロードされた
**プログラム＋結果＋レポート**）を**閲覧して承認**する。

> **改訂（#42, N3 / #54）**: #41 で提出が「プログラム＋結果アップロード」型へ変わったため、旧審査
> （提出物一覧から合成結果フラグメントを読み、承認で生データ結果を並置）は撤去した。**生データ
> 並置（合成 vs 生）の核メッセージを並置する解説面は UI に設けず**、原理は提出物（プログラム＋
> 結果＋レポート）の閲覧・承認という審査の流れの中で示す（#54）。

> **デモ用プリセット（#53）**: 一覧／審査が初回から空でないよう、起動時に**仮想（プリセット）
> 提出物**を seed する（§7.8 #53）。これらは `submitted`/`approved` を混在し、既存の審査ビューで
> そのまま閲覧・承認できる（**架空の成果物**。in-memory）。

### 審査の起点（提出物一覧から選ぶ）

- 個別ページの「紐づく提出物」一覧で、各提出物カードに **owner 限定の審査導線**（`review-open`）を
  出す。非 owner（analyst）には審査導線を出さず、ガード注記（`review-guard`）を表示する
  （**擬似ガード**）。
- 審査導線を押すと**審査ビュー**（`review-view`）が開く。

### 提出物の閲覧（プログラム＋結果＋レポート）

- 審査ビューは、選んだ提出物の以下を表示する（共有ストア `submissions.find()` の実体を読む）。
  - **プログラム（コード）**: `program.text` を `<pre><code x-text>` で表示（**実行しない**）。
  - **結果**: `results[]` を順に表示。`type:'image'` は提出時の **dataURL を `<img :src>`** で、
    `type:'text'` は `<pre x-text>` で表示。
  - **レポート**: `report`（あれば）。提出者・日時も表示。
- **生データ並置（合成 vs 生）は行わない**（`#sub-review-*` の raw フラグメント読込・`.compare`
  は審査ビューに存在しない）。表示はすべて `x-text`／提出時 dataURL に閉じ、`htmx.ajax` や
  `x-html` は使わない。

### 状態遷移（status: submitted → approved）

- 「承認する」で提出物の `status` を **`submitted → approved`** に遷移させる。更新は共有ストア
  `Alpine.store('submissions').approve(dataset_id, submission_id, reviewer)` 経由で、対象提出物の
  実体を直接書き換える（`reviewedBy` / `reviewedAt` も記録）。
- in-memory なので個別ページの提出物一覧の `submission-status` 表示が **approved に即時反映**
  される（カードに `data-status="approved"` も付く）。承認済み提出物を再度開くと
  プログラム＋結果が復元表示される（承認ボタンは無効化）。

### owner 限定ガード（仮定）

- 審査は **owner ロールのみ**可能（`canReview() = role === 'owner'`）。
- **仮定**: catalog の `owner` 名と突き合わせる厳密な所有権一致は擬似のため簡略化し、
  「owner ロールなら（このデータセットの提出物を）審査できる」とした。本番では TRE/サーバーが
  データセット所有権でサーバー側認可を強制する（UI ガードは UX 用）。

### 本番との対応

| 関心事 | モック（本リポジトリ） | 本番プロトタイプ |
|--------|------------------------|------------------|
| 提出物の閲覧 | アップロードしたプログラム（`x-text`）＋結果（dataURL/テキスト） | DB から取得しレビュー UI で表示 |
| プログラム適用 | 行わない（表示のみ） | TRE 内で同一プログラムを生データへ隔離実行 |
| 審査状態 | Alpine 共有ストアの `status`（in-memory） | サーバー側ジョブ状態 + DB |
| 審査認可 | owner ロール限定の擬似ガード | サーバー側で所有権により強制 |
| 合成≈生の核メッセージ | 提出→審査の体験で示す（並置する解説面は持たない） | 解説／オンボーディング面 |

## 7.10 合成 vs 生の核メッセージの扱い（#54）

「同一コードが合成・生で動き、結果は近いが非同一」という**核メッセージ**（§1）は、UI 上に
**合成 vs 生を左右に並置する解説面（原理デモ面）を設けない**（#54 で撤去）。原理は、合成データの
ダウンロード → プログラム＋結果の提出 → オーナーによる審査・承認という**実コンテンツの体験**の
流れの中で説明する。

- データカタログ（一覧・個別ページ）・提出・審査のどのビューにも合成 vs 生の並置は出さない。
- 審査ビューの承認 callout では「本番では TRE 内で同じプログラムを生データへ適用し、本物の
  レポートを作成する」旨を文章で説明する（並置や専用面への導線は持たない）。
- 生データ（`raw`）は TRE 隔離の擬似のため UI からは一切リンクしない。analyst フラグメント
  （合成結果）の事前生成は維持し（`generator/`）、将来本番 htmx へ移植できる形を保つ。

### 生成物との関係

- `generator/` のフラグメント生成（`fragments/<id>/owner/*`・`fragments/<id>/analyst/*`）は
  **維持する**（UI からの参照有無に関わらず生成物として保持）。本変更は **UI のみ**の撤去で、
  ジェネレータ・データ・フラグメント生成のロジックは変更しない。

## 8. 合成データ生成方針（mockdata-generator の思想）

`gghatano/mockdata-generator` の仕様駆動アプローチ（source → generator → synthetic +
evaluation）を参照しつつ、本デモでは簡易ジェネレータで代替する。

- **source**: 臨床的素性（指数減衰・用量依存 nadir・リスク依存の進行）をパラメータ化
- **generator**: シード固定で再現可能に生成（`generate_data.py`）
- **synthetic**: 別シード + 分布シフトで生成し、生データと近いが非同一にする
- **evaluation**: 3分析を両データに適用し、結果が「近いが一致しない」ことを UI 上で比較

### 8.1 複数データセット（#22）

`DATASETS` レジストリで複数データセットを定義する。各データセットは raw/synthetic の
`Profile`（seed・shift・`id_prefix`・`risk_params`）を持ち、シード固定で再現生成する。

- `prostate-psa`: 既存の前立腺がん PSA データセット（seed/パラメータは不変、出力バイト一致）。
- `renal-marker`: 第 2 データセット（腎細胞がんの腫瘍マーカー）。**PSA と同一スキーマ**
  （`risk_group` / `psa`(=腫瘍マーカー値) / `drug,dose_mg`）を保ち、別ドメイン・別 seed・
  別パラメータで生成する。これにより既存 3 分析（`analyses.py`）を**変更せずそのまま**
  適用できる。

> 仮定（#22）: 3 分析は PSA スキーマに結合しているため、第 2 データセットも同一スキーマで
> 別ドメインを表現する設計とした。分析ロジック・分布シフト方針・既存 seed は変更しない。

## 9. 検証方法

```bash
uv sync        # 依存を取得（pyproject.toml / .python-version）
make build     # site/data/*.json と site/fragments/**/*.html を生成
make serve     # http://localhost:8000/
```

- ① データカタログ → 一覧・個別ページで合成データ由来のダミーデータと活用例を確認
- ② 分析者 → 合成データ（JSON）をダウンロードし、プログラム＋結果を提出
- ③ オーナー審査 → 提出物（プログラム＋結果＋レポート）を閲覧し、承認で `approved` へ遷移
- `uv run python -m generator.generate_data` を再実行してデータが再現生成できる（シード固定）
